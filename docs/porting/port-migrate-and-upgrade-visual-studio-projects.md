# <a name="project-migration-and-upgrade-reference-for-visual-studio"></a>Dokumentacja migracji i uaktualniania projektu dla programu Visual Studio

::: moniker range="vs-2017"

Każda wersja programu Visual Studio zwykle obsługuje większość poprzednich typów projektów, plików i innych zasobów. Można z nich korzystać, [gdy zawsze się znajdują](../ide/solutions-and-projects-in-visual-studio.md), i pod warunkiem, że nie zależą od nowszych funkcji, program Visual Studio próbuje zachować zgodność wstecz z poprzednimi wersjami, takimi jak Visual Studio 2015, Visual Studio 2013 i visual Studio 2012. (Zobacz [Informacje o wersji](/visualstudio/releasenotes/vs2017-relnotes/) , dla których funkcje są specyficzne dla których wersji).

Obsługa niektórych typów projektów zmienia się również w miarę upływu czasu. Nowsza wersja programu Visual Studio może nie obsługiwać już niektórych projektów lub wymaga aktualizacji projektu, tak aby nie był już zgodny z poprzednią wersją. Bieżący stan problemów dotyczących migracji można znaleźć w [witrynie społeczności deweloperów programu Visual Studio](https://aka.ms/feedback/suggest?space=8).

Ten artykuł zawiera szczegółowe informacje dotyczące typów projektów, które program Visual Studio 2017 może migrować. Artykuł wyklucza typy projektów, które nie są już obsługiwane w programie Visual Studio 2017 i dlatego nie można przeprowadzić migracji. Artykuł wyklucza również obsługiwane typy projektów, które nie mają problemów z migracją; Ta lista jest dostępna na [platformie docelowej i zgodności](/visualstudio/productinfo/vs2017-compatibility-vs).

> [!IMPORTANT]
> Niektóre typy projektów wymagają zainstalowania odpowiednich obciążeń za pomocą Instalatora programu Visual Studio. Jeśli nie masz zainstalowanego obciążenia, program Visual Studio zgłosi nieznany lub niezgodny typ projektu. W takim przypadku Sprawdź opcje instalacji i spróbuj ponownie. Aby uzyskać szczegółowe informacje o obsłudze projektu w programie Visual Studio 2017, zobacz artykuł dotyczący [platformy i zgodności](/visualstudio/productinfo/vs2017-compatibility-vs) .

## <a name="how-visual-studio-decides-when-to-migrate-a-project"></a>Jak program Visual Studio decyduje o czasie migracji projektu

Każda nowa wersja programu Visual Studio zwykle poszukuje zgodności z poprzednimi wersjami, dzięki czemu ten sam projekt może być otwarty, modyfikowany i zbudowany w różnych wersjach. Jednak czasami zmiany są zmieniane w miarę upływu czasu, ponieważ niektóre typy projektów nie mogą być już obsługiwane. (Zobacz [ukierunkowane na platformę i zgodność](/visualstudio/productinfo/vs2017-compatibility-vs) , dla których typy projektów są obsługiwane w programie Visual Studio 2017). W takich przypadkach nowsza wersja programu Visual Studio nie będzie ładować projektu i nie oferuje ścieżki migracji. należy zachować ten projekt w poprzedniej wersji programu Visual Studio, która go obsługuje.

W innych przypadkach nowsza wersja programu Visual Studio może otworzyć projekt, ale należy zaktualizować lub zmigrować projekt w taki sposób, aby był niezgodny z poprzednimi wersjami. Program Visual Studio używa wielu kryteriów, aby określić, czy taka migracja jest konieczna:

- Zgodność z docelowymi wersjami platform z powrotem do Visual Studio 2013 RTM.

- Zgodność zasobów czasu projektowania z poprzednimi wersjami programu Visual Studio. (Mianowicie różne kanały programu Visual Studio 2017; Visual Studio 2015 RTM & Update 3; Visual Studio 2013 RTM & Update 5; Visual Studio 2012 Update 4; Visual Studio 2010 z dodatkiem SP 1) Program Visual Studio 2017 ma na celu bezproblemowy niepowodzenie pracy z przestarzałymi zasobami czasu projektowania bez ich uszkodzenia, dzięki czemu poprzednie wersje nadal mogą otworzyć projekt.

- Czy nowe zasoby czasu projektowania spowodują uszkodzenie zgodności z poprzednimi wersjami do Visual Studio 2013 RTM & Update 5.

Właściciel inżyniera danego typu projektu przegląda te kryteria i wykonuje połączenie, w którym dotyczy to obsługi, zgodności i migracji. Ponownie program Visual Studio próbuje zachować przezroczystą zgodność między wersjami programu Visual Studio, jeśli jest to możliwe, co oznacza, że jedna z nich może tworzyć i modyfikować projekty w jednej wersji programu Visual Studio, a sama działa w innych wersjach.

Jeśli jednak taka zgodność nie jest możliwa, podobnie jak w przypadku niektórych typów projektów opisanych w tym artykule, program Visual Studio otworzy kreatora uaktualniania, aby wprowadzić niezbędne zmiany jednokierunkowe.

Takie jednokierunkowe zmiany mogą wymagać zmiany `ToolsVersion` właściwości w pliku projektu, co oznacza, że wersja MSBuild może przekształcić kod źródłowy projektu na możliwy do uruchomienia i możliwe do wdrożenia artefakty. To znaczy, co renderuje projekt niezgodny z poprzednimi wersjami programu Visual Studio, nie jest wersją programu *Visual Studio* , ale wersją programu *MSBuild* określoną przez `ToolsVersion` . Tak długo, jak Twoja wersja programu Visual Studio zawiera łańcucha narzędzi MSBuild, który jest zgodny z `ToolsVersion` w projekcie, program Visual Studio może wywołać ten łańcucha narzędzi w celu skompilowania projektu.

Aby zachować maksymalną zgodność z projektami utworzonymi w starszych wersjach, program Visual Studio 2017 zawiera niezbędną łańcuchy narzędzi MSBuild do obsługi `ToolsVersion` 15, 14, 12 i 4. Projekty, które używają dowolnej z tych `ToolsVersion` wartości, powinny spowodować pomyślne skompilowanie. (Temat ponownie, aby program Visual Studio 2017 obsługiwał typ projektu, zgodnie z opisem w temacie [Określanie wartości docelowej platformy i zgodność](/visualstudio/productinfo/vs2017-compatibility-vs)).

W tym kontekście pytanie naturalnie się pojawia, niezależnie od tego, czy należy próbować ręcznie zaktualizować lub zmigrować projekt do nowszej `ToolsVersion` wartości. Wprowadzenie takiej zmiany jest niepotrzebne i prawdopodobnie wygeneruje wiele błędów i ostrzeżeń, które należy poprawić, aby projekt mógł zostać ponownie skompilowany. Ponadto, jeśli program Visual Studio porzuca obsługę określonego `ToolsVersion` w przyszłości, otwarcie projektu spowoduje wyzwolenie procesu migracji projektu, ponieważ `ToolsVersion` należy zmienić wartość. W takim przypadku podsystem dla danego typu projektu wie dokładnie, co należy zmienić, i może wprowadzać te zmiany automatycznie zgodnie z opisem we wcześniejszej części tego artykułu.

## <a name="migrate-a-project"></a>Migrowanie projektu

Chociaż staramy się zachować zgodność z poprzednimi wersjami, mogą zostać wprowadzone zmiany, które nie są zgodne z poprzednimi wersjami. (Zobacz [ukierunkowane na platformę i zgodność](/visualstudio/releases/2019/compatibility) , dla których typy projektów są obsługiwane w programie Visual Studio 2019). W takim przypadku nowsza wersja programu Visual Studio nie będzie ładować projektu ani nie oferuje ścieżki migracji. Może być konieczne zachowanie tego projektu w poprzedniej wersji programu Visual Studio.

Czasami nowsza wersja programu Visual Studio może otworzyć projekt, ale musi zaktualizować lub migrować projekt w taki sposób, aby był niezgodny z poprzednimi wersjami. Program Visual Studio używa wielu kryteriów, aby określić, czy taka migracja jest konieczna:

- Zgodność z docelowymi wersjami platform z powrotem do Visual Studio 2013 RTM.

- Zgodność zasobów czasu projektowania z poprzednimi wersjami programu Visual Studio. (Mianowicie różne kanały programu Visual Studio 2019, Visual Studio 2017; Visual Studio 2015 RTM & Update 3; Visual Studio 2013 RTM & Update 5; Visual Studio 2012 Update 4; Visual Studio 2010 z dodatkiem SP 1) Program Visual Studio 2019 ma na celu bezproblemowy niepowodzenie pracy z przestarzałymi zasobami czasu projektowania bez ich uszkodzenia, dzięki czemu poprzednie wersje nadal mogą otworzyć projekt.

- Czy nowe zasoby czasu projektowania spowodują uszkodzenie zgodności z poprzednimi wersjami do Visual Studio 2013 RTM & Update 5.

Zespół inżynieryjny, który jest właścicielem tego typu projektu, analizuje te kryteria i wykonuje połączenie, w którym dotyczy to obsługi, zgodności i migracji. Firma Microsoft spróbuje ponownie zachować zgodność między wersjami programu Visual Studio, aby podczas tworzenia i modyfikowania projektów w jednej wersji programu Visual Studio działała ona tylko w innych wersjach.

Czasami zgodność nie jest możliwa. Następnie program Visual Studio Otwiera Kreatora uaktualniania, aby wprowadzić niezbędne zmiany jednokierunkowe. Te jednokierunkowe zmiany mogą wymagać zmiany `ToolsVersion` właściwości w pliku projektu, co oznacza, że wersja MSBuild może przekształcić kod źródłowy projektu w możliwy do uruchomienia i możliwe do wdrożenia artefakty.

Co renderuje projekt niezgodny z poprzednimi wersjami programu Visual Studio, nie jest wersją programu *Visual Studio* , ale wersją programu *MSBuild* określoną przez `ToolsVersion` . Jeśli Twoja wersja programu Visual Studio zawiera łańcucha narzędzi MSBuild, który jest zgodny z `ToolsVersion` w projekcie, program Visual Studio może wywołać ten łańcucha narzędzi w celu skompilowania projektu.

Aby zachować zgodność z projektami utworzonymi w poprzednich wersjach, program Visual Studio 2019 zawiera niezbędną łańcuchy narzędzi MSBuild do obsługi `ToolsVersion` 15, 14, 12 i 4. Projekty, które używają dowolnej z tych `ToolsVersion` wartości, powinny spowodować pomyślne skompilowanie. (Temat ponownie, aby program Visual Studio 2019 obsługiwał typ projektu, zgodnie z opisem w temacie [Określanie wartości docelowej platformy i zgodność](/visualstudio/releases/2019/compatibility)).

Może być konieczne ręczne zaktualizowanie lub przeprowadzenie migracji projektu do nowszej `ToolsVersion` wartości. Nie jest konieczne dokonanie takiej zmiany i prawdopodobnie wygeneruje wiele błędów i ostrzeżeń, które należy naprawić, aby umożliwić ponowne skompilowanie projektu. Ponadto, jeśli program Visual Studio nie obsługuje określonego `ToolsVersion` w przyszłości, projekt wyzwala proces migracji projektu podczas jego otwierania, ponieważ jego `ToolsVersion` wartość musi zostać zmieniona.
